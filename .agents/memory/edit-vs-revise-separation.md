---
name: Edit vs Revise separation
description: How to let a plain "edit settings" endpoint recalculate derived data without ever touching a separate append-only history/log table, even when a naive rebuild would normally rewrite it.
---

When a resource has both (a) a plain "edit settings" endpoint and (b) a dedicated "create a timeline event" endpoint (e.g. Edit Tenant vs. Rent Revision, or any base-value + revision-log pattern), the two must stay fully independent: the edit endpoint must be allowed to run unconditionally (never rejected because history exists) and must never insert, update, or delete a history row — not even "automatic/system-generated" ones.

**Why:** First attempt made rebuild-on-edit skip only when the edited field matched a rejection guard (400 if the value differed from the timeline). The user explicitly rejected that: editing must always succeed and never be blocked by pre-existing history. Second attempt still failed review because the generic "full rebuild" path (used for lease dates, escalation settings, etc.) deleted and regenerated automatic history rows as a side effect — an architect review caught this live (automatic revision amounts changed after an unrelated field edit).

**How to apply:** Don't try to detect "is this specific field safe to edit" — instead make the shared rebuild/recalculation routine take an explicit `preserveRevisionHistory` (or similarly named) flag. When true: skip every delete/insert/update against the history table, and skip whatever helper normally audits/backfills those rows — derived amounts must still be computed correctly by reading the append-only log + the live settings directly (never by reading the "automatic" audit rows, which should be pure output, not input, to any calculation). The plain edit endpoint always calls the rebuild with the flag set; only the dedicated revision-creating endpoint(s) call it unflagged (or omit the flag). This makes "edit never touches history" a structural guarantee instead of a per-field guard that's easy to miss for a newly added billing-defining field.

Also watch for: a denormalized "current value" column (e.g. `tenant.rentAmount`) that only acts as the source of truth when zero history rows exist — once history exists, the column is just a stored snapshot and the API response should keep showing the live-computed value from the timeline, not the raw column, even right after a successful edit to that column. This can look like "the edit didn't take" if not explained.

**Superseded design note:** the paragraph above described treating the denormalized column as authoritative only pre-history. A later redesign made the column *always* authoritative for "current" (never overridden/derived by the timeline, never resynced from it), while the timeline stays authoritative for historical-period pricing. See [Current-value column vs event timeline promotion](current-value-timeline-promotion.md) for how to implement that split without reintroducing coupling.
