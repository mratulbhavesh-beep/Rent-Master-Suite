==================================================

payments.amount must remain the single source of truth for:
"How much money was actually received."

Never modify a payment amount merely because its allocation changes.

payment_allocations must only represent:
"How the received money was applied to rent periods."

Reports must continue to use the actual payment received date/month/year as currently designed.

Do NOT change report accounting from cash-received date to allocation-period accounting.

Example:
If ₹12,000 is received in July but allocated to January and February rent periods, it must still appear in the July payment/cash-flow report.

PHASE 3 — FIFO & Specific Period Allocation Engine
-------------------------

Implement the planned server-side payment allocation engine.

Support:

1. AUTO / FIFO allocation
   - Allocate to the oldest unpaid generated rent period first.
   - Then continue to the next unpaid period.
   - Support partial payments.
   - Support one payment spanning multiple periods.

2. SPECIFIC PERIOD allocation
   - User can select one generated rent period.
   - Apply payment only to that selected period.
   - If payment exceeds the remaining due of the selected period, the excess must become Advance Credit.
   - Do NOT automatically forward that excess to another period in Specific Period mode.

3. Pending Adjustment
   - If a valid target period does not yet exist where applicable, preserve the payment as Pending Adjustment according to the approved design.
   - When the appropriate period is later generated, adoption must happen safely without duplicating the payment or allocation.

4. Payment Edit
   - Editing a payment must safely clear and rebuild affected allocations.
   - Payment Edit is HIGH RISK.
   - The entire edit operation must run in ONE database transaction.
   - Payment update + old allocation removal + new allocation creation + affected rent status recomputation must succeed or fail together.
   - If any step fails, rollback the entire transaction.

5. Payment Delete
   - Safely remove affected allocation rows.
   - Recompute affected generated rent period statuses.
   - Deleting one of multiple payments on the same period must correctly recalculate paid/partial/pending status.
   - Do not re-trigger adoption for a deleted payment.

6. Add the planned billing-period endpoint.

7. Update OpenAPI/codegen if required.

8. Add all planned API and rent-calculation tests.

Do not change the working Advance/Post-paid rent-generation engine while implementing FIFO.
